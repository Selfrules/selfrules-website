# Plan 06-03: Performance Audit and Optimization

## Objective

Verify and optimize the site to meet all Lighthouse targets (95+ on Performance, Accessibility, Best Practices, SEO) and bundle size constraints (< 100KB first load JS, CLS < 0.05, LCP < 2.5s). This plan runs AFTER SEO metadata and accessibility/animations are in place so measurements reflect the final state.

## Tasks

<task id="06-03-01">
<title>Build analysis and bundle size verification</title>
<read_first>
- package.json (scripts section)
- next.config.ts or next.config.mjs (current config)
- src/app/globals.css (check total CSS)
</read_first>
<action>
1. Run `npx next build` and capture the output.
2. Check the build output table for:
   - First Load JS size for each route -- must be < 100KB (PERF-05)
   - All routes should show circle icon (static), not lambda (dynamic)
   - No build warnings or errors
3. If any route exceeds 100KB first load JS:
   - Check which Client Components are contributing most JS
   - Verify only 4-5 Client Components exist: navbar, mobile-menu, language-toggle, scroll-reveal, (optionally count-up-number and blinking-cursor)
   - Check if any unnecessary `'use client'` directives exist on pages or sections
   - Remove any unused imports or dependencies
4. If any route is dynamic (lambda icon):
   - Verify `setRequestLocale(locale)` is called in every layout and page
   - Verify `generateStaticParams` is exported from layouts with [locale] param
   - Check for accidental usage of dynamic APIs (cookies(), headers())
</action>
<acceptance_criteria>
- npx next build completes with exit code 0
- Build output shows all routes with static (circle) icon
- First Load JS column shows < 100KB for every route
- No build warnings about dynamic rendering
</acceptance_criteria>
</task>

<task id="06-03-02">
<title>CLS prevention audit</title>
<read_first>
- src/lib/fonts.ts (font configuration)
- src/app/[locale]/layout.tsx (font variable application)
- src/components/sections/hero.tsx (above-the-fold content)
- .planning/phases/06-seo-polish-deploy/06-UI-SPEC.md (CLS Prevention section)
</read_first>
<action>
Verify CLS prevention measures are in place (PLSH-04, target < 0.05):

1. **Fonts**: Verify `next/font/google` is used with `display: 'swap'` for Inter, Space Grotesk, and JetBrains Mono. The `adjustFontFallback: true` (default) generates size-matched fallback metrics.

2. **Above-the-fold content**: Verify the Hero section has no dynamic content injection. All content should be statically rendered.

3. **ScrollReveal initial state**: Verify `.animate-fade-up-initial` sets `opacity: 0` but does NOT affect layout (uses `transform: translateY` not `margin` or `height`). This prevents sections from affecting document flow.

4. **Navbar height**: Verify the navbar has a fixed/known height and the main content has corresponding padding-top to prevent shift when navbar transitions from transparent to opaque.

5. **No images above fold**: The site has no images in v1 (the hero is typography-only), so no image CLS issues.

6. If any CLS-causing patterns are found, fix them:
   - Add `min-height` to sections if needed during hydration
   - Ensure font variables are applied to `<html>` or `<body>` so they're available immediately
   - Ensure no JavaScript injects content above the fold after initial render
</action>
<acceptance_criteria>
- grep "display.*swap" src/lib/fonts.ts confirms font display swap setting
- grep "animate-fade-up-initial" src/app/globals.css confirms transform-only (no layout-affecting) initial state
- The Hero section renders immediately without depending on client-side JS
</acceptance_criteria>
</task>

<task id="06-03-03">
<title>Lighthouse audit and targeted fixes</title>
<read_first>
- Build output from task 06-03-01
- src/app/globals.css (accessibility CSS)
- src/app/[locale]/layout.tsx (metadata, lang attribute)
</read_first>
<action>
1. Start the production server: `npx next build && npx next start`
2. Run Lighthouse audit (or document the manual steps):
   - Open `http://localhost:3000` in Chrome
   - Open DevTools > Lighthouse
   - Run audit for: Performance, Accessibility, Best Practices, SEO
   - Target: 95+ on all four categories

3. Common fixes if scores are below 95:

   **Performance < 95:**
   - Check if unused CSS is bloating the page (unlikely with Tailwind purge)
   - Verify no synchronous scripts block rendering
   - Check Umami script has `async defer` attributes

   **Accessibility < 95:**
   - Missing alt text on images (shouldn't apply -- no images in v1)
   - Missing form labels (shouldn't apply -- no forms)
   - Color contrast failures -- check text-tertiary usage (from task 06-02-08)
   - Missing landmark roles (fixed in task 06-02-06)

   **Best Practices < 95:**
   - Mixed content warnings (shouldn't apply with HTTPS)
   - Deprecated APIs (shouldn't apply with modern stack)
   - Console errors

   **SEO < 95:**
   - Missing meta description (fixed in plan 06-01)
   - Missing hreflang (fixed in plan 06-01)
   - Robots.txt blocking resources
   - Non-crawlable links

4. Document any issues found and fix them. Re-run Lighthouse after fixes.
</action>
<acceptance_criteria>
- Lighthouse Performance score >= 95 (documented in build verification output or screenshot)
- Lighthouse Accessibility score >= 95
- Lighthouse Best Practices score >= 95
- Lighthouse SEO score >= 95
- No critical Lighthouse warnings remain unfixed
</acceptance_criteria>
</task>

<task id="06-03-04">
<title>Core Web Vitals verification</title>
<read_first>
- Build output from task 06-03-01
- Lighthouse report from task 06-03-03
</read_first>
<action>
Verify Core Web Vitals meet targets (PERF-06):

1. **LCP (Largest Contentful Paint) < 2.5s**:
   - For a static site with no images, LCP is typically the largest text element (hero headline)
   - Static rendering + edge CDN (Vercel) should make this trivially fast
   - If LCP > 2.5s, check: font loading delay, render-blocking resources, server response time

2. **CLS (Cumulative Layout Shift) < 0.1 (target < 0.05 per PLSH-04)**:
   - Verified in task 06-03-02
   - If CLS > 0.05, investigate font swap, navbar transition, ScrollReveal initial states

3. **INP (Interaction to Next Paint)**:
   - Minimal interactivity in this site (nav clicks, language toggle)
   - Should be well under 200ms target

4. Check these values in Lighthouse report (from task 06-03-03) under the Performance section metrics.

5. If any metric is outside target, investigate and fix the specific cause.
</action>
<acceptance_criteria>
- LCP < 2.5s as reported by Lighthouse
- CLS < 0.1 (ideally < 0.05) as reported by Lighthouse
- No Core Web Vitals flagged as "poor" in Lighthouse report
</acceptance_criteria>
</task>
